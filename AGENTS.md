# AGENTS.md

本文件定义本项目中 Codex / Claude / 其他代码代理的自主工作规则。代理应优先阅读并遵守本文件，然后再阅读 `docs/` 下的项目文档。

## 最高优先级指令

你可以在不频繁询问用户的情况下持续自主修改代码，只要这些修改是本地的、可回退的，并且符合项目文档。

优先实现并执行 `docs/bdd.md` 中的 BDD 规则。当需求不完全明确时，根据现有文档和代码推断最自洽的实现。只有在操作具有破坏性、影响外部系统、需要重大架构选择，或需求之间直接冲突时，才停下来询问用户。

每轮修改后运行相关测试，修复失败，然后继续处理下一个最有价值的改进点。

## 项目目标

本项目的目标是构建一个硬规则驱动的多视角小说世界模拟 / 生成系统。

代码和生成流程必须优先保证：

- 世界规则自洽。
- 角色行为符合自身信息、目标和资源。
- 主角没有剧情豁免。
- 图谱、章节、事件之间因果可追溯。
- 所有生成结果可被 BDD 验收规则检查。
- 修改后系统仍可运行、可测试、可继续扩展。

## 自主执行范围

代理可以自主执行以下操作，不需要询问用户：

- 阅读、分析、重构现有代码。
- 新增必要的函数、类、模块、测试。
- 修复运行错误、类型错误、lint 错误。
- 根据已有文档补全缺失实现。
- 根据 BDD 规则设计校验器、数据结构、工作流。
- 拆分过大的文件或函数。
- 删除确认无用的死代码。
- 优化命名、结构和可读性。
- 多轮运行测试并修复失败。
- 在不改变目标的前提下调整实现方案。

代理不需要因为以下普通工程选择打断用户：

- 文件该放在哪里，只要项目结构能推断出来。
- 函数名、类名、变量名的普通命名选择。
- 使用哪种内部实现方式，只要符合现有技术栈。
- 是否补测试：只要修改影响行为，就应该补。
- 是否修复由本次修改引入的错误：应该直接修。
- 是否运行相关测试：应该直接运行。
- 是否整理局部重复代码：应该直接整理。

## 必须询问用户的情况

只有遇到以下情况时，代理才应该停下来询问用户：

- 需要删除大量文件、重置代码、覆盖用户未提交修改。
- 需要执行破坏性命令，例如 `rm -rf`、`git reset --hard`。
- 需要修改数据库、线上服务、外部 API 或共享资源。
- 需要安装、删除、降级核心依赖。
- 需求本身存在明显矛盾，无法合理推断。
- 有多个架构方向，且选择会显著影响长期项目形态。
- 当前文档与代码严重冲突，无法判断哪一个是权威来源。
- 需要生成大量内容但缺少关键业务设定，例如世界观、境界体系、角色档案。

除此之外，代理应自行判断并继续推进。

## 决策优先级

修改代码时，按以下顺序决策：

1. 正确性。
2. 与 `docs/bdd.md` 的规则一致。
3. 可测试性。
4. 可维护性。
5. 简洁性。
6. 性能。
7. 风格统一。

如果简洁性和可解释的因果链冲突，优先选择因果链清晰。

如果快速实现和未来可验证冲突，优先选择可验证。

## 信息来源优先级

当不同信息来源冲突时，按以下顺序判断：

1. 用户当前明确指令。
2. 项目中的 BDD、spec、design 文档。
3. 现有测试。
4. 现有代码结构。
5. 通用最佳实践。
6. 代理自己的推断。

如果 `docs/bdd.md` 与普通代码习惯冲突，优先遵守 `docs/bdd.md`。

## BDD 驱动开发规范

任何与小说生成、图谱构建、章节生成、事件生成、角色决策相关的代码，都必须能映射到 BDD 场景。

### 图谱构建必须支持

- 角色目标。
- 角色资源。
- 角色认知视图。
- 信息来源。
- 信息可靠性。
- 信息危险等级。
- 认知层级。
- 硬规则。
- 资源消耗。
- 时间消耗。
- 因果链。
- `COSTS` / `CONSUMES` / `KNOWS` / `BELIEVES` 等关系。

如果某个数据结构无法表达这些内容，代理应主动扩展它。

### 章节生成必须检查

- 角色是否只基于自身信息行动。
- 角色是否使用了可用资源。
- 决策是否服务长期目标。
- 主角是否被规则豁免。
- 越级胜利是否有不可复制因素。
- 机缘是否有来源、稀缺性和代价。
- 信息危险是否兑现。
- 不可靠信息是否带不确定性。
- 视角叙事是否以当前视角角色为中心。

如果当前实现没有校验流程，代理应主动补一个验证层。

## 数据模型规范

### 命名原则

使用业务语义命名，不使用抽象空词。

推荐命名：

- `CharacterPerspective`
- `KnowledgeNode`
- `CausalChain`
- `HardRule`
- `ResourcePool`
- `ValidationReport`
- `ChapterDraft`
- `PerspectiveView`
- `CognitiveLayer`

避免命名：

- `Data`
- `Info`
- `Manager`
- `Processor`
- `Handler`
- `Thing`
- `Obj`
- `Temp`

除非项目已有明确约定，否则不要使用过度泛化的名字。

### 角色相关模型

角色至少应能表达：

- `id`
- `name`
- `core_desire`
- `long_term_goal`
- `current_goal`
- `resources`
- `known_information`
- `beliefs`
- `cognitive_layer`
- `power_level`
- `constraints`
- `relationships`

角色的客观状态和主观视图必须分离。

不允许让角色直接访问全局真相，除非该角色在图谱中确实 `KNOWS` 该信息。

### 信息相关模型

信息节点至少应能表达：

- `id`
- `content`
- `source`
- `reliability`
- `danger_level`
- `cognitive_layer`
- `known_by`
- `distortion_rules`
- `expressible_in_lower_layers`

跨认知层传递信息时，必须发生衰减、变形或失败。

### 规则相关模型

硬规则必须显式标记。

推荐字段：

- `id`
- `name`
- `type`
- `condition`
- `effect`
- `is_hard`
- `exceptions`

如果 `is_hard = true`，任何角色触发条件都必须承受结果，包括主角。

## 函数设计规范

函数应该小而明确。

推荐函数名：

- `validateVillainDecision`
- `traceCausalChain`
- `verifyResourceBalance`
- `applyCognitiveDecay`
- `buildPerspectiveView`
- `checkHardRuleViolation`
- `generateValidationReport`

避免一个函数同时做：

- 读取输入。
- 修改图谱。
- 调用模型。
- 生成正文。
- 校验规则。
- 写入文件。

如果函数超过一个明确职责，代理应主动拆分。

## 校验器规范

所有校验器应返回结构化结果，不要只返回 `true` / `false`。

推荐结构：

```ts
type ValidationResult = {
  passed: boolean
  scenarioId: string
  failures: ValidationFailure[]
  evidence: Evidence[]
}
```

失败信息必须说明：

- 哪个规则失败。
- 哪个对象失败。
- 为什么失败。
- 需要什么证据才能通过。
- 是否需要重写。

不要返回模糊错误，例如 `Invalid chapter.`。

应该返回类似：

```text
B.1 failed: protagonist defeated a higher-level opponent without causal chain evidence.
```

## 生成流程规范

章节或事件生成不得一步到位直接输出正文。

推荐流程：

1. 构建客观图谱。
2. 构建角色主观视图。
3. 生成事件候选。
4. 追踪因果链。
5. 执行 BDD 校验。
6. 对失败项重写。
7. 生成章节草稿。
8. 再次校验。
9. 输出最终结果与 validation report。

如果当前系统没有分层，代理应逐步拆出来。

## 测试规范

凡是修改核心逻辑，必须优先补测试。

测试应覆盖：

- 正常通过场景。
- 违反 BDD 的失败场景。
- 主角触发硬规则。
- 反派使用已有资源。
- 信息跨层衰减。
- 资源消耗守恒。
- 越级战斗失败或付出代价。
- 多视角叙事差异。

测试名称要表达业务行为，而不是实现细节。

推荐：

- `villain_does_not_overestimate_protagonist_without_information`
- `hard_rule_applies_to_protagonist`
- `breakthrough_consumes_shared_resource`
- `knowledge_decays_across_cognitive_layers`

避免：

- `test_case_1`
- `test_validate`
- `test_run`

## 错误处理规范

只在系统边界做防御性校验：

- 用户输入。
- 文件读取。
- 外部 API。
- LLM 返回结果。
- JSON 解析。
- 数据库读写。

内部函数不要堆无意义 fallback。

不要吞错误。

禁止：

```ts
try {
  // ...
} catch {
  return null
}
```

应该保留足够上下文：

```ts
throw new ValidationError("Missing causal chain for victory event", {
  eventId,
  scenarioId: "B.1",
})
```

## LLM 调用规范

如果项目使用 LLM 生成内容，代理应遵守：

- Prompt 必须明确输入、输出、约束。
- 输出必须尽量结构化。
- 生成正文前应先生成可校验中间结构。
- 不信任 LLM 输出，必须经过校验器。
- 不把“模型说合理”当作通过依据。
- 失败时根据 validation report 重写，而不是盲目重试。

LLM 生成内容必须服从图谱和 BDD，不允许反过来修改规则迎合生成结果。

## 文件组织规范

推荐结构：

```text
docs/
  bdd.md
  coding-guidelines.md

src/
  domain/
    character/
    knowledge/
    rule/
    graph/
    chapter/

  validation/
    scenarios/
    validators/
    reports/

  generation/
    graph/
    chapter/
    perspective/

  tests/
```

如果项目已有结构，代理应优先沿用现有结构，不要强行重排。

只有当现有结构阻碍实现时，才进行小步重构。

## 重构规范

代理可以主动重构，但必须遵守：

- 不为好看做大规模重构。
- 不改变外部行为，除非任务要求。
- 每次重构都要让代码更容易验证 BDD。
- 重构后必须运行相关测试。
- 不同时做多个互不相关的大改动。

推荐重构方向：

- 拆分超大函数。
- 分离客观图谱与角色主观视图。
- 分离生成与校验。
- 把散落的规则集中到 validator。
- 把魔法字符串变成枚举或常量。
- 把隐式状态变成显式数据结构。

## 注释规范

默认不写注释。

只有在以下情况写简短注释：

- 解释非显然的业务规则。
- 解释 BDD 中的硬约束。
- 解释为什么不能简化某段逻辑。
- 解释认知层衰减、不可知层等复杂机制。

不要写代码做了什么的注释。

禁止：

```ts
// loop through characters
```

允许：

```ts
// Hard rules apply before protagonist-specific narrative adjustments.
```

## 提交前自检

每完成一轮修改，代理应自动检查：

- 是否破坏已有测试。
- 是否新增必要测试。
- 是否违反 `docs/bdd.md`。
- 是否引入主角豁免。
- 是否让角色获得不该知道的信息。
- 是否让资源、时间、实力凭空变化。
- 是否把生成逻辑和校验逻辑耦合过深。
- 是否存在未使用代码。
- 是否存在临时调试输出。

如果发现问题，代理应继续修复，不要先询问用户。

## 持续工作模式

当用户说“继续”、“一直改”、“自己推进”、“不用问我”时，代理应按以下循环工作：

1. 阅读当前文档和代码。
2. 找到与目标差距最大的部分。
3. 制定局部修改目标。
4. 修改代码。
5. 补测试。
6. 运行测试。
7. 修复失败。
8. 总结本轮变化。
9. 继续下一轮。

只有遇到必须询问用户的情况才停下。

## 禁止行为

代理不允许：

- 为了让测试通过而降低 BDD 要求。
- 为了生成爽点而绕过硬规则。
- 让主角获得未解释的特殊豁免。
- 让反派因为剧情需要降智。
- 让资源、时间、境界凭空变化。
- 把校验失败改成 warning 来逃避失败。
- 盲目添加 fallback 掩盖数据错误。
- 大规模重写无关代码。
- 删除用户未确认的重要内容。
- 执行破坏性命令。
- 在没有授权时操作远程仓库或外部服务。

## 输出规范

每轮完成后，输出应简洁说明：

- 改了什么。
- 为什么改。
- 测试结果。
- 下一步准备做什么。

不要输出大段完整代码，除非用户要求。

推荐格式：

```text
已完成本轮修改：

- 增加了 CharacterPerspective，用于分离角色主观视图和客观图谱。
- 增加了 A.1 / A.2 的校验器，防止反派基于作者视角决策。
- 补充了失败用例，覆盖“反派无理由不用资源”的情况。
- 测试已通过。

下一步我会继续实现 B 组实力与主角豁免相关校验。
```
